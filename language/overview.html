TemScript

Background

Near the end of my undergraduate studies, I was contemplating whether or not to enroll into graduate school and get my Masters. When I was learning more towards doing it, I considered what my thesis topic would be. The choice I was learning the most towards was designing my own programming language. While I didn't have any idea what it would do, how it would be written, or what the overall goal was, I still thought that that would be the most interesting topic for me. I eventually decided to enter the job market rather than get Masters at that time (Currently, I'm in the middle of getting my Masters in Computer Science).

Recently, I been reading articles and listening to videos regarding programming in general. Doing this has sparked my interest in making my own programming language again. Although it isn't required for my Masters degree (the Masters program that I'm doing doesn't require a thesis at all), I still feel motivated to attempt this. First, I want to break down the goals and rules of the language. This will be done by listening the things I like and dislike about other programming langauges.

Goals

  • Allow for multiple paradigms
    • Procedural
    • Fuctional
    • OOP
  • Simplify abstraction and encapsulation
  • Reduce overall code that needs to be written
  • Must be easily capable for being modeled

Abilities

  • Must be able to transpile to other (popular) programming langauges
    • C#
    • F#
    • TypeScript
    • Java
    • Rust
    • More?

Rules

  • Functions
    • Functions must be first-class types
    • All functions must be pure
    • Functions may not take a class as a parameter
  • Structs
    • All structs must implement an interface. Any struct that doesn't have an interface implemented will have one created and implemented automatically by the transpiler.
    • There are no access modifiers for struct fields
    • There are no struct methods. Only extensions for interfaces.
    • There are no constructors.
    • All associated struct values must be fields unless decorated with an attribute. Then, it is a property
    • Fields in structs can be decorated with attributes to define how the property setter/getter should be defined
  • Interfaces
    • Interfaces can only have readonly values defined. No methods
    • Interfaces can have "method-like" calls similar to how extensions work in C#
    • Interfaces automatically extend other interfaces that have the same fields
